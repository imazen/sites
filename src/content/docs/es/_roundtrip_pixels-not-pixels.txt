### Device pixels ≠ browser pixels ≠ image pixels

Thanks to Apple, CSS and HTML pixels (`px`) **are usually mapped to multiple device/printer pixels**. My phone's `devicePixelRatio` is 3.875, [what's yours?](https://www.mydevice.io/)

My desktop `devicePixelRatio` changes with the browser zoom level, but it's set to 1.5 by default (my `System > Display > Scale` is 150% after all).

Image files (excluding SVGs) are decoded into a grid of pixels. **It's extremely difficult to make an image's pixels display perfectly in a 1:1 relationship with device pixels, but with `srcset` you can get close enough**, and a little bit of JavaScript can adjust the size of the HTML element after loading if you truly have a justified reason for needing that perfect mapping, although such forbidden artifacts will not be revealed here.

> Other absolute units ([in, pc, pt, mm, cm, Q](https://developer.mozilla.org/en/docs/Web/CSS/length)) are defined in terms of a number of CSS pixels, and assumed to be 96dpi. Developers avoid absolute units because it's generally better to use units that derive from the font-size or viewport/container dimensions. Incompatible units can break a design if the user adjusts their `Accessibility > Text size` multiplier, zooms in the browser, or rotates their device. Of course, you should do all of these things before launching your site.
