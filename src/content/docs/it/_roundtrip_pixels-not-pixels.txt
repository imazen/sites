### Device Pixels ≠ Browser px ≠ Image px

Thanks to Apple, CSS and HTML pixels (`px`) nearly always map to **device/printer pixels in multiples**. My phone's devicePixelRatio is 3.875; [what's yours?](https://www.mydevice.io/)

The desktop's devicePixelRatio changes during browser zooming, but default is 1.5 (my `System > Display > Scale` is 150%, after all).

Image files (excluding SVG) are decoded into pixel grids. **It's annoyingly difficult to get images' pixels to visually match device pixels 1:1, but with `srcset` you can get close enough**, and some JavaScript can adjust the HTML element size after load if you have a good reason for a perfect mapping, though these dark arts won't be divulged here.

> Other [absolute lengths (in, pc, pt, mm, cm, Q)](https://developer.mozilla.org/en-US/docs/Web/CSS/length) are defined in terms of a CSS pixel count that pretends everything is 96 dpi. Developers avoid absolute units because it is generally better to use sizes relative to font size or viewport/container dimensions. Unmatched units can break layout if the user adjusts the `Accessibility > Text size` multiplier, zooms the browser, or rotates the device. Needless to say, you should test all these things before distributing your site.
