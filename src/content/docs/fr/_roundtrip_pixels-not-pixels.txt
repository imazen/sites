### Device pixels ≠ browser pixels ≠ image pixels

Thanks to Apple, CSS and HTML pixels (`px`) **are almost always mapped to multiple device/printer pixels**. `devicePixelRatio` on my phone is 3.875; [what about yours?](https://www.mydevice.io/)

My desktop computer's `devicePixelRatio` changes with browser zoom, but it is 1.5 by default (because my `System > Display > Scale` setting is at 150%).

Image files (excluding SVGs) decode to a grid of pixels. **It is annoyingly difficult to display a pixel of the image perfectly at a 1:1 ratio with the device pixels, but with `srcset`, you can get a reasonable approximation**, and a little bit of javascript can adjust the size of the HTML element after loading if you have a truly justifiable reason for wanting this perfect mapping, although these dark arts will not be unveiled here.

> Other absolute units [(in, pc, pt, mm, cm, Q)](https://developer.mozilla.org/en-US/docs/Web/CSS/length#absolute_lengths) are defined in terms of the number of CSS pixels and pretend that everything is 96dpi. Developers avoid absolute units because it is generally better to use units derived from font size or viewport/container dimensions. Disparate units can break layout if the user adjusts their `Accessibility > Text size` multiplier, zooms the browser, or rotates their device. It goes without saying that you should do all of these things before publishing your site.
